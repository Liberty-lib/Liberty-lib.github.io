详细分析

Base64

开头一大堆print略过，直接来到用户输入发现被写入到v51中。往下走发现经过了trim_matches，通过ai得知这只是去掉多余字符，对数据没影响。往下跟踪v39，往下看有个很明显的base64。

Ascii85

下面一个很大的for循环实则看起来很吓人，实际只是在执行Ascii85，这里稍微展开介绍一下Ascii85的特征。

特征 1：出现“85”这个常量（最强指纹）

题中是这一句：

if ( v24 >= 0x55 ) // 0x55 = 85 core::panicking::panic_bounds_check(v24, 85LL, ...);

特征 2：有一个长度=85 的“字符表”，并用 table[index] 映射

v25 = a0_0[v24];

a0_0 为 0x21..0x75（'!'..'u'）
题中引用的正好是Ascii85 标准的字母表。

特征 3：4 字节一组处理 + 最多输出 5 个字符

题里两处很直观：

  • “每次前进 4” 的痕迹（原样）：
    v10 = v7 + 4; v16 = v53 + v7;

  • “最多 5 个字符”的边界检查（原样）：
    if ( j > 4 ) core::panicking::panic_bounds_check(j, 5LL, ...);
    这对应 Ascii85 的基本规律：
    4 个输入字节（32 位）→ 5 个 Ascii85 字符。

特征 4：一坨“除以 85 / 取余数”的计算（常见被编译器优化成乘法+右移）

题里这段就是把一个 32 位值拆成 5 个 base85 “位”：
v21 = (3233857729LL * v20) >> 38; LOBYTE(v51[1]) = v20 - 85 * v21; …`

BytesToHexChars

读题不仔细，本以为这题就到这结束了，但发现flag不对，后来在学长提醒下发现漏了个函数没读
<alloc::string::String as core::iter::traits::collect::FromIterator<char>>::from_iter(v40, v54, v9, v11);
打开这个函数，可以发现明显的BytesToHexChars，在字符串窗口中也能很明显的发现 0123456789abcdef
